Perché l’autenticazione è diventata uno standard, non un “login”

All’inizio bastavano una form, una password e una sessione sul server. Poi sono arrivati API, mobile app, SPA, microservizi, cloud e integrazioni tra aziende. Il problema non era più solo “chi sei?”, ma anche “chi si fida di chi?”, “chi può fare cosa?” e “come lo dimostro senza regalare password in giro?”.

Idea chiave

Gli standard sono nati perché i sistemi hanno smesso di essere un’unica applicazione chiusa.

OAuth, OpenID Connect, JWT e PKCE non sono “moda enterprise”: sono risposte a problemi pratici di delega, identità, trust, browser, mobile e API.

1. Il problema originale

login → sessione → cookie

Come funzionava agli inizi

Una web app monolitica aveva quasi tutto nello stesso posto: pagine, logica di business, database utenti e sessioni. Il browser mandava username e password, il server verificava le credenziali e salvava uno stato di sessione.

Utente

Inserisce username e password nella form di login.

→

Server monolitico

Controlla le credenziali e crea una sessione lato server.

→

Cookie di sessione

Il browser conserva solo un identificatore, non tutta l’identità.

Perché funzionava

Il perimetro era piccolo

Il server era l’unico “controllore”. La stessa applicazione mostrava le pagine, decideva i permessi e parlava col database. Il cookie era come il timbro sul polso all’ingresso di un locale: rientri e il buttafuori riconosce che eri già passato.

Contesto ideale: una sola web app, stesso dominio, pochi server, utenti locali, niente app mobile, niente API esterne, niente “login con Google”, niente microservizi.

2. La rottura del modello

da monolite a ecosistema
Cosa cambia

Non c’è più un solo server che decide tutto

Il web diventa distribuito. Le applicazioni parlano con altre applicazioni. Il browser non è più l’unico client. Le API devono essere chiamate da mobile app, SPA, processi batch, partner esterni e microservizi.

1

Credenziali sparse

Ogni sistema chiede password, le duplica e aumenta la superficie di attacco.

2

Trust tra sistemi

Un’API deve capire se fidarsi di un client che non è il browser originale.

3

Scalabilità

Le sessioni lato server diventano più difficili da gestire su più nodi e domini.

4

Delega

Un’app deve agire “per conto dell’utente” senza conoscere la sua password.

3. Timeline evolutiva

causa → soluzione → nuovo limite
problema precedente soluzione introdotta limite residuo
anni 90 → oggi
01

Sessioni e cookie

Nascono per dare memoria a HTTP, che di base è stateless: ogni richiesta è indipendente dalle precedenti.

PrimaIl server non ricordava chi aveva già fatto login.
SoluzioneSessione lato server + cookie con session id nel browser.
LimitiCSRF, session fixation, furto cookie, problemi multi-dominio e scaling.
HTTP auth
02

Basic Auth

Semplice e utile per scenari tecnici basilari: il client manda credenziali nell’header Authorization a ogni richiesta.

PrimaServiva un modo standard per autenticare richieste HTTP senza form HTML.
SoluzioneUsername e password codificati Base64, da usare solo sopra TLS.
LimitiNessuna delega, nessuno scope, password ripetuta spesso, logout debole.
2010
03

OAuth 1.0 / 1.0a

Nasce per evitare il pattern pericoloso “dammi la tua password così accedo io al tuo account”. Introduce l’idea moderna di accesso delegato.

PrimaLe app terze chiedevano direttamente la password dell’utente.
SoluzioneL’utente autorizza una terza app senza consegnarle le credenziali.
LimitiFirme crittografiche complesse, implementazioni fragili, scarsa ergonomia per mobile e browser moderni.
2012 → draft 2.1
04

OAuth 2.0 → OAuth 2.1

OAuth 2.0 rende la delega più flessibile: access token, refresh token, scope, authorization server, resource server. OAuth 2.1 consolida le pratiche sicure più moderne, come Authorization Code + PKCE, e rimuove pattern deprecati o rischiosi.

PrimaOAuth 1.0 era sicuro ma macchinoso e difficile da implementare bene.
SoluzioneFramework di autorizzazione delegata per web app, API, mobile e servizi.
LimitiNon è autenticazione da solo; redirect URI, token lifetime, scope e client type vanno capiti bene.
OpenID → OIDC
05

OpenID → OpenID Connect

OAuth risponde a “cosa può fare questo client?”. OpenID Connect aggiunge una risposta standard a “chi è l’utente?”. Per questo introduce l’ID token e claim di identità interoperabili.

PrimaMolti usavano OAuth 2.0 per il login, ma OAuth non definiva l’identità utente.
SoluzioneIdentity layer sopra OAuth 2.0: ID token, issuer, subject, audience, nonce, UserInfo.
LimitiClaim mapping, provider diversi, verifica token e logout federato restano complessi.
2015 → oggi
06

JWT

JWT non è “un sistema di login”: è un formato compatto per trasportare claim. Può essere firmato e verificato da più servizi senza consultare sempre il server che lo ha emesso.

PrimaAPI e microservizi avevano bisogno di informazioni portabili sull’utente o sul client.
SoluzioneToken firmato con claim leggibili: subject, issuer, audience, scadenza, permessi.
LimitiRevoca difficile, token troppo grandi, leakage, validazione errata di firma/issuer/audience/exp.

4. Perché servono tutti questi standard

interoperabilità + sicurezza
01

Autenticazione ≠ autorizzazione

Autenticazione: “chi sei?”. Autorizzazione: “cosa puoi fare?”. Confonderle porta a sistemi fragili: login riuscito non significa accesso a tutto.

02

Delega dell’identità

Come una delega notarile: non consegni le chiavi di casa, autorizzi qualcuno a fare una specifica cosa, per un tempo limitato.

03

Identity Provider centrale

Un IdP concentra login, MFA, password policy, passkey, account lifecycle e auditing. Le app consumano token invece di reinventare login.

04

Single Sign-On

L’utente accede una volta presso un provider fidato e poi entra in più applicazioni senza ridigitare credenziali ovunque.

05

Trust distribuito

API e microservizi verificano token firmati, issuer, audience, scadenza e claim. Il trust non è “mi fido perché arriva una chiamata”.

Analogia: in un piccolo ufficio basta riconoscersi di persona. In un aeroporto internazionale servono passaporto, boarding pass, controlli, gate, aree riservate e regole comuni tra compagnie diverse.

6. Cosa risolve ogni tecnologia

mappa mentale operativa
OAuth
Problema risolto

Autorizzazione delegata: una terza app può accedere a una risorsa senza conoscere la password dell’utente.

Da non confondere con

Non è, da solo, un protocollo di autenticazione utente.

OIDC
Problema risolto

Autenticazione federata: standardizza il login sopra OAuth 2.0 tramite ID token e claim di identità.

Attenzione

Verificare sempre issuer, audience, firma, scadenza e nonce quando previsto.

JWT
Problema risolto

Trasporto compatto di informazioni firmate: claim utili a API e microservizi.

Attenzione

Un JWT rubato spesso è utilizzabile fino alla scadenza; non metterci dati sensibili inutili.

PKCE
Problema risolto

Protegge i client pubblici dall’intercettazione del codice di autorizzazione.

Attenzione

Non sostituisce TLS, validazione redirect URI, hardening browser/mobile e corretta gestione token.

7. Limiti e complessità introdotte

non sempre più standard = più sicurezza
Perché oggi è più difficile

La sicurezza si è spostata dalla password alla configurazione

Molte violazioni moderne non dipendono da “OAuth è insicuro”, ma da integrazioni sbagliate: token accettati dall’audience sbagliata, redirect URI permissive, secret committati, scope troppo larghi, refresh token trattati come caramelle.

!
Configurazione complessa

Issuer, audience, metadata, signing keys, clock skew, scope, claim mapping e lifetime devono combaciare.

!
Overengineering

Per una piccola app server-rendered, OIDC enterprise, microservizi e JWT ovunque possono essere più rischiosi che utili.

!
Token leakage

Access token in localStorage, log, query string o crash report possono trasformarsi in credenziali riutilizzabili.

!
Confusione semantica

Scope, role e permission non sono la stessa cosa. Usarli a caso rompe l’autorizzazione.

Quando non serve complicare

Il modello semplice ha ancora senso

Non ogni progetto deve partire da OAuth/OIDC/JWT. Serve scegliere in base al rischio reale.

✓ App server-side monolitica con utenti locali e nessuna API pubblica: cookie auth + sessione possono bastare.
✓ Tool interno piccolo, dietro rete aziendale e con pochi utenti: evitare federazione complessa se non richiesta.
✓ Prototipo non esposto: mantenere semplice, ma senza salvare password in chiaro o disabilitare TLS.
✓ API solo backend-to-backend controllata: valutare client credentials, mTLS o meccanismi gestiti dal cloud.

Regola pratica: usa lo standard quando risolve un problema reale di trust, delega, federazione o distribuzione. Non usarlo solo perché “fa enterprise”.

8. Come si inserisce .NET

ASP.NET Core 10
Il framework incapsula, ma non sostituisce la comprensione

ASP.NET Core ti dà handler, middleware e policy: tu devi sapere cosa stai configurando

.NET non “inventa” un modo proprietario di autenticare: espone astrazioni coerenti per integrare cookie, JWT bearer, OpenID Connect, Identity Provider esterni, policy di autorizzazione e claim-based authorization.

Web app server-side

Di solito usa cookie per la sessione applicativa e OIDC per autenticare presso un Identity Provider.

AddCookie()AddOpenIdConnect()[Authorize]

Web API

Di solito valida access token JWT emessi da un authorization server: firma, issuer, audience, scadenza e claim.

AddJwtBearer()AuthorityAudience

Autorizzazione

La decisione “può farlo?” si esprime con policy, requirement, claim, role e handler custom, non solo con il fatto che l’utente sia loggato.

AddAuthorization()PolicyClaims

Cosa deve capire uno sviluppatore .NET: differenza tra cookie e bearer token, tra ID token e access token, tra authentication scheme e authorization policy, tra role/scope/permission, e perché i token vanno validati completamente prima di fidarsi.

Perché .NET incapsula

Ridurre errori ripetitivi

Parsing token, discovery metadata, challenge, sign-in, sign-out, claim mapping e validazione sono attività ripetitive e delicate. Gli handler riducono codice fragile.

Cosa resta a te

Scelte di architettura

Decidere se usare cookie, BFF, OIDC, JWT bearer, client credentials, policy custom e dove conservare i token dipende dal tipo di client e dal rischio.

Errore tipico

“Ho un JWT, quindi sono sicura”

No. Un JWT è sicuro solo se emesso da un issuer fidato, firmato correttamente, destinato alla tua API, non scaduto e trattato come credenziale.
