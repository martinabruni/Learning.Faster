Come funziona .NET: dal codice alla CPU

In un’app .NET classica, il codice C#, F# o VB viene compilato in IL/CIL più metadati. Il runtime carica l’assembly, il CLR gestisce esecuzione, sicurezza, type system, threading e memoria; il JIT traduce i metodi in codice macchina quando servono. In alternativa, con Native AOT parte del lavoro viene anticipato in fase di publish.

source

Codice sorgente

C#, F# o VB. Il compilatore lavora contro il target framework, ad esempio net10.0.

→
build

SDK & compilatore

Il .NET SDK compila, risolve NuGet, genera assembly e produce IL + metadata.

→
IL/CIL

Intermediate Language

Codice intermedio portabile, non ancora codice macchina specifico della CPU.

→
runtime

CLR + JIT + GC

Il runtime carica, verifica, ottimizza, compila a codice nativo ed esegue.

SDK vs Runtime

SDK Serve per sviluppare: CLI, build, restore, publish, template, analyzer.
Runtime Serve per eseguire: CLR, GC, JIT, base class libraries.
ASP.NET Runtime Runtime specializzato per app web, API, middleware e hosting.
Workload Estensioni per MAUI, iOS, Android, WASM, Aspire e scenari specifici.

Dentro il CLR

JIT compiler Converte IL in codice macchina on-demand e applica ottimizzazioni.
Garbage Collector Gestisce heap, generazioni, oggetti vivi e rilascio automatico memoria.
Type system Garantisce metadata, type safety, reflection e interoperabilità tra linguaggi.
ThreadPool & async Supporta I/O asincrono, concorrenza e workload server ad alta scala.

Limiti reali

Dove .NET 10 è meno competitivo

.NET 10 è forte, ma non è la scelta migliore per ogni contesto. I limiti dipendono soprattutto da ecosistema, startup/footprint, UI multipiattaforma, browser e lock-in culturale verso Microsoft/Azure.

🐍

Contro Python: AI/data science puro

.NET oggi ha astrazioni AI buone per applicazioni enterprise, ma Python resta dominante per notebook, ricerca ML, librerie scientifiche, prototipazione data e community AI.

Meno librerie ML native Meno standard nei notebook
🟨

Contro Node.js: web full-stack leggero

JavaScript/TypeScript resta naturale nel browser e nel frontend. Node vince quando il team vuole un solo linguaggio tra frontend, SSR, tooling web e funzioni leggere.

Browser non nativo Bundle WASM più pesanti
☕

Contro Java: ecosistema JVM storico

In organizzazioni già standardizzate su Spring, JVM, Kafka tooling e librerie Java, .NET può essere meno conveniente per migrazioni greenfield solo parziali.

Switching cost Ecosistemi enterprise diversi
🐹

Contro Go: servizi piccoli e binari semplici

Go resta molto competitivo per CLI, microservizi minimali, tool di infrastruttura e deployment con binario statico semplice. Native AOT riduce il gap, ma non copre ancora tutti gli scenari .NET.

AOT non universale Footprint da valutare
Nota tecnica importante Native AOT migliora startup, memoria e distribuzione, ma impone vincoli su reflection, dynamic loading, generics, trimming e compatibilità framework. In ASP.NET Core, alcune feature sono pienamente supportate, altre parziali o non supportate: quindi va progettato dall’inizio, non “acceso alla fine”.

✅

Scegli .NET 10 se

Devi costruire backend robusti, API performanti, sistemi enterprise, integrazioni cloud/Azure, servizi osservabili e prodotti mantenibili da team strutturati.

Valutalo bene se
