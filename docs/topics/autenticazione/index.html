Idea chiave: autenticazione = “chi sei?”. Autorizzazione = “cosa puoi fare?”. Sono collegate, ma non sono la stessa cosa.
1. Intro

Autenticazione e autorizzazione non vanno confuse.

In una web app moderna non basta sapere che un utente “ha fatto login”. Devi anche sapere quali risorse può usare, da quale client arriva, quanto è affidabile il token e quali rischi stai introducendo.

Autenticazione

Verifica l’identità

Risponde alla domanda: “Chi è questo utente?”. Può produrre una sessione, un cookie, un ID token o un insieme di claim.

LoginPasswordlessMFAOIDC
Autorizzazione

Verifica i permessi

Risponde alla domanda: “Cosa può fare?”. Usa ruoli, policy, scope, claim e regole applicative per decidere l’accesso.

RolePolicyClaimScope
Perché conta

Le app sono distribuite

Frontend, API, microservizi, app mobile e servizi cloud parlano tra loro. Serve un modo standard per trasportare identità e permessi.

Perché è fragile

I token sono credenziali

Un access token rubato può essere usato come una chiave. Per questo durata, storage, validazione e revoca sono fondamentali.

Errore comune

Fidarsi del browser

Il client può essere manipolato. Il backend deve sempre validare token, audience, issuer, firma, scadenza e permessi.

3. Standard fondamentali

Gli standard da conoscere come developer .NET.

OAuth 2.1, OIDC, JWT, PKCE e refresh token non sono intercambiabili. Ogni pezzo ha un compito preciso. Confonderli porta a bug di sicurezza.

OAuth 2.1

Autorizzazione delegata

OAuth 2.1 è l’evoluzione di OAuth 2.0: consolida pratiche più sicure, spinge Authorization Code Flow con PKCE e rimuove pattern storicamente rischiosi. A maggio 2026 è ancora un draft IETF, non un RFC finale.

A cosa serve

Dà a un’app accesso limitato a una risorsa senza consegnarle la password dell’utente.

Come funziona

  1. Il client manda l’utente all’Authorization Server.
  2. L’utente si autentica e concede il consenso.
  3. Il client riceve un authorization code.
  4. Il client scambia il code con un access token.
  5. L’API usa il token per decidere se concedere accesso.
OpenID Connect

Identità sopra OAuth

OIDC è uno strato di identità costruito sopra OAuth 2.0. Permette al client di verificare l’identità dell’utente e ricevere claim standardizzati.

A cosa serve

Serve quando non vuoi solo autorizzare l’accesso a una risorsa, ma vuoi sapere chi è l’utente.

Come funziona

  1. Il client richiede scope come openid, profile, email.
  2. L’Identity Provider autentica l’utente.
  3. Il client riceve un ID token.
  4. Il client valida firma, issuer, audience, nonce e scadenza.
  5. I claim diventano identità applicativa.
JWT

Formato compatto di token

JWT è un formato compatto e URL-safe per trasferire claim tra due parti. Può essere firmato e, in alcuni casi, cifrato.

A cosa serve

Rappresenta informazioni come subject, audience, issuer, scope, ruoli e scadenza.

Come funziona

  1. Header: indica tipo e algoritmo.
  2. Payload: contiene i claim.
  3. Signature: protegge integrità e autenticità.
  4. Il backend valida firma e claim critici.
  5. Il backend non deve accettare token solo perché “sembra un JWT”.
PKCE

Protezione del code flow

PKCE mitiga l’attacco di intercettazione dell’authorization code ed è ormai centrale nei flussi moderni.

A cosa serve

Impedisce che un attaccante rubi il code e lo scambi al posto del client legittimo.

Come funziona

  1. Il client genera un code_verifier casuale.
  2. Calcola un code_challenge, di solito con SHA-256.
  3. Manda il challenge nella richiesta di login.
  4. Quando scambia il code, manda il verifier originale.
  5. Il server confronta challenge e verifier prima di emettere token.
Refresh Token

Rinnovo della sessione

Un refresh token serve a ottenere nuovi access token dopo la scadenza. È molto sensibile perché può prolungare l’accesso nel tempo.

A cosa serve

Evita di chiedere il login a ogni scadenza dell’access token.

Come funziona

  1. L’Identity Provider emette access token breve e refresh token più duraturo.
  2. Quando l’access token scade, il client usa il refresh token.
  3. Il token endpoint restituisce un nuovo access token.
  4. Con rotation, viene emesso anche un nuovo refresh token.
  5. Se viene rilevato riuso sospetto, la sessione può essere invalidata.
Cookie vs Token

Sessione server o token bearer?

Cookie-based auth e token-based auth risolvono problemi simili, ma con superfici di rischio diverse.

A cosa servono

I cookie sono comodi per web app server-side. I bearer token sono pratici per API, app mobile, SPA e architetture distribuite.

Come funzionano

  1. Cookie auth: il server crea una sessione o un ticket protetto.
  2. Il browser rimanda automaticamente il cookie a ogni richiesta.
  3. Token auth: il client invia esplicitamente il token nell’header.
  4. Il backend valida il token a ogni chiamata.
  5. Cookie richiede attenzione a CSRF; token richiede attenzione a theft e storage.
Nota aggiornata: OAuth 2.1 è importante da conoscere, ma a maggio 2026 va trattato come draft. Per production, usa provider e librerie che seguono OAuth 2.0, PKCE, OIDC e le best current practice di sicurezza.
4. Come funziona in .NET

ASP.NET Core usa middleware, scheme, handler e policy.

In .NET configuri scheme di autenticazione, handler per cookie/JWT/OIDC, e policy di autorizzazione applicate su endpoint e controller.

Mini diagramma del flusso in ASP.NET Core

1. Request HTTP

Arriva una chiamata con cookie o header Authorization.

2. Authentication middleware

Legge lo scheme, invoca l’handler e costruisce ClaimsPrincipal.

3. Authorization middleware

Applica policy, role, claim e scope prima di eseguire l’endpoint.

Authentication middleware

Capisce “chi sei”

ASP.NET Core Authentication usa scheme e handler. Esempi: cookie handler, JWT bearer handler, OpenID Connect handler.

builder.Services
    .AddAuthentication("Bearer")
    .AddJwtBearer("Bearer", options =>
    {
        options.Authority = "https://login.microsoftonline.com/{tenantId}/v2.0";
        options.Audience = "api://my-api";
    });

app.UseAuthentication();
app.UseAuthorization();
Authorization

Decide “cosa puoi fare”

L’autorizzazione usa policy, ruoli, claim e scope. Prima si identifica l’utente, poi si valutano i permessi.

builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("CanReadInvoices", policy =>
        policy.RequireAuthenticatedUser()
              .RequireClaim("scope", "invoices.read"));
});

app.MapGet("/invoices", () => "ok")
   .RequireAuthorization("CanReadInvoices");
ASP.NET Core Identity

Gestisce utenti locali

Identity aggiunge login UI, gestione utenti, password, ruoli, conferma email, reset password e integrazione con Entity Framework.

Entra ID, Auth0, Keycloak

Per scenari enterprise, SSO e identità centralizzata, una web app ASP.NET Core può integrarsi con Microsoft Entra ID tramite OAuth 2.0 e OpenID Connect.

Regola pratica .NET: usa cookie/OIDC per web app server-side, JWT bearer per API, Identity per utenti locali, Microsoft.Identity.Web/MSAL per integrazioni Microsoft Entra ID.
5. Flusso completo

Step by step: dal login all’accesso alla risorsa.

Questa sequenza è il modello mentale da ricordare. Le implementazioni cambiano, ma i passaggi concettuali restano simili.

1

Login utente

L’utente parte dal client e clicca “Accedi”.

2

Redirect a IdP

Il client manda l’utente all’Identity Provider.

3

Emissione token

Il provider restituisce code e poi token.

4

Validazione API

Il backend verifica firma, issuer, audience, scadenza.

5

Accesso risorse

Solo policy e scope validi passano.

Cosa vede il client

  • Una pagina di login o un redirect.
  • Un token da usare nelle chiamate API.
  • Una scadenza dopo cui rinnovare la sessione.

Cosa deve fare il backend

  • Validare sempre il token lato server.
  • Non fidarsi di claim modificabili dal client.
  • Applicare autorizzazione su ogni endpoint sensibile.
6. Rischi di sicurezza

La parte critica: gli standard non salvano da configurazioni sbagliate.

Molte vulnerabilità nascono perché gli standard vengono usati male: token salvati male, redirect aperti, claim non validati, CORS permissivo, autorizzazione solo lato frontend.

Token theft

Furto di token

Spiegazione

Un token rubato può essere usato come credenziale bearer.

Esempio

Access token salvato in localStorage e letto da script malevolo.

Mitigazione

Token brevi, storage sicuro, HttpOnly cookie, rotazione, revoca, HTTPS.

XSS

Cross-Site Scripting

Spiegazione

Un attaccante esegue JavaScript nel browser dell’utente.

Esempio

Commento HTML non sanificato che legge dati sensibili.

Mitigazione

Output encoding, CSP, sanitizzazione, niente token accessibili via JS.

CSRF

Cross-Site Request Forgery

Spiegazione

Il browser invia automaticamente cookie a un sito su cui l’utente è loggato.

Esempio

Una pagina malevola forza una POST verso la tua app cookie-based.

Mitigazione

Anti-forgery token, SameSite, controlli origin/referer.

Replay attack

Riuso di una richiesta o token

Spiegazione

Un attaccante riusa un token o una richiesta valida già catturata.

Esempio

Token intercettato in log o proxy e riutilizzato prima della scadenza.

Mitigazione

HTTPS, token brevi, nonce, jti, sender-constrained token dove disponibile.

Misconfigured auth

Autenticazione configurata male

Spiegazione

La configurazione accetta token che non dovrebbe accettare.

Esempio

API che valida la firma ma non controlla audience o issuer.

Mitigazione

Validare issuer, audience, lifetime, firma, algoritmo e policy.

Over-trusting client

Fidarsi troppo del frontend

Spiegazione

Il frontend è ambiente ostile: può essere letto, modificato e automatizzato.

Esempio

Il pulsante “admin” è nascosto via CSS, ma l’endpoint non è protetto.

Mitigazione

Autorizzazione sempre server-side, deny-by-default, test sugli endpoint.

Regola d’oro: il client può migliorare UX, ma non deve essere la barriera di sicurezza. La barriera vera sta nel backend, nelle policy e nell’Identity Provider.
7. Limiti e trade-off

Nessuna tecnologia è sempre la scelta giusta.

Scegliere bene significa capire quando una tecnologia è adatta, quali rischi porta e quando introduce più complessità che valore.

Tecnologia
Pro
Contro
Quando NON usarla
JWT
Compatto, stateless, adatto ad API e microservizi, contiene claim verificabili.
Revoca difficile, rischio leak, payload leggibile se non cifrato, claim obsoleti fino a scadenza.
Non usarlo come sessione lunga piena di dati sensibili o quando ti serve revoca immediata semplice.
OAuth 2.x
Standard per accesso delegato, scope, consenso, accesso API senza password condivise.
Non è autenticazione pura; implementazioni errate creano redirect, token e scope vulnerabili.
Non usarlo da solo per “login” senza OIDC; non usarlo per saltare autorizzazione backend.
OIDC
Standardizza login federato, ID token, claim, discovery e integrazione con Identity Provider.
Più componenti da configurare: redirect URI, nonce, issuer, client, callback, logout.
Non serve se hai solo comunicazione machine-to-machine senza identità utente.
Cookie auth
Ottima per web app server-side, HttpOnly, Secure, SameSite, integrazione naturale col browser.
Il browser invia cookie automaticamente: rischio CSRF se non mitigato.
Non è ideale per API pubbliche multi-client o app mobile che non usano browser session-based.
8. Best practice per dev .NET

Checklist pratica: cosa fare e cosa evitare.

Usa questa lista quando configuri una web app, una API o un’integrazione con Entra ID.

Da fare
✓

Usa HTTPS ovunque

Token e cookie devono viaggiare solo su connessioni cifrate.

✓

Preferisci Authorization Code Flow + PKCE

È il pattern moderno per applicazioni con utente interattivo.

✓

Valida sempre i JWT lato API

Firma, issuer, audience, scadenza, algoritmo e claim richiesti.

✓

Usa policy-based authorization

Le policy sono più espressive dei soli ruoli e riducono duplicazioni.

✓

Proteggi i cookie

Imposta HttpOnly, Secure, SameSite e anti-forgery dove serve.

✓

Fai log sicuro

Non loggare access token, refresh token, authorization code o cookie di sessione.

Da evitare
×

Non salvare token sensibili in localStorage se puoi evitarlo

In caso di XSS diventano facilmente leggibili da JavaScript.

×

Non fidarti di claim lato frontend

Il frontend può nascondere o mostrare UI, ma non decide i permessi reali.

×

Non accettare token con audience sbagliata

Un token emesso per un’altra API non deve valere per la tua.

×

Non usare wildcard nei redirect URI

Redirect troppo permissivi facilitano furti di code e token.

×

Non mettere segreti in SPA o app mobile

Un client pubblico non può proteggere davvero un client secret.

×

Non proteggere solo controller “visibili”

Ogni endpoint sensibile deve avere autorizzazione server-side esplicita.

Template mentale per scegliere in .NET

Web app MVC/Razor

Cookie + OIDC

Login federato, sessione browser protetta, token gestiti server-side.

Web API

JWT Bearer
