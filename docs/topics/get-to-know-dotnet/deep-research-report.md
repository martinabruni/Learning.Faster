## Platform map you should internalize first

The most important conceptual distinction is between the platform and the app models built on top of it. **.NET** is the platform: SDK, runtime, CLI, base libraries, host, tooling, and support policy. **ASP.NET Core** is the web framework on top of .NET. **Blazor** is a .NET UI model within ASP.NET Core for interactive web UI. **.NET MAUI** is the cross-platform native UI stack for mobile and desktop. **.NET Framework** is the older Windows-only implementation that still matters mainly for existing applications. Microsoft also documents that target frameworks such as `net8.0`, `net9.0`, and `net10.0` are how projects declare the API surface they compile against, while workloads extend the SDK for optional app models such as MAUI. ŅłĆciteŅłéturn24search6Ņłéturn24search0Ņłéturn24search1Ņłéturn24search15Ņłéturn25search2Ņłéturn26search0Ņłéturn24search14Ņłü

A second distinction that many juniors blur is **SDK vs runtime**. The SDK is what you install to build: it includes the CLI, build logic, templates, and a runtime. The runtime is what you install to run framework-dependent apps. Microsoft also notes that the SDK version constrains which target frameworks you can build for, which is why upgrading to a new .NET version usually starts with installing a newer SDK, not just changing a TFM in the project file. ŅłĆciteŅłéturn24search16Ņłéturn24search2Ņłéturn24search12Ņłéturn24search7Ņłü

A third distinction is **support train vs feature excitement**. LTS matters because production systems live longer than conference demos. STS is fine when a team consciously accepts a shorter support window and has a clear upgrade motion. For serious study, learn on .NET 10 first, compare with .NET 8 where relevant, and understand .NET 9 mainly as a bridge release that introduced capabilities you may still encounter in active codebases. ## Concepts people use all the time without really understanding

A **Host** is the object responsible for startup and lifetime management and the binding point for DI, configuration, and logging. In modern .NET, that concept sits underneath far more app types than many juniors realize. A **Middleware** is a request delegate in an ordered pipeline; changing order changes behavior. A **Service lifetime** is not a registration ornament but a statement about ownership and reuse semantics. A **DbContext** is a unit-of-work object, not a global repository. ŅłĆciteŅłéturn1search20Ņłéturn9search1Ņłéturn1search13Ņłéturn10search1Ņłü

`IQueryable` and `IEnumerable` may look syntactically similar in LINQ, but they are not semantically interchangeable. `IEnumerable` means in-memory iteration over objects already in process. `IQueryable` means ŌĆ£this query may be interpreted by a provider,ŌĆØ which is why EF Core can translate query shapes to SQL. A **Task** is not a thread. A **Thread** is an execution resource. A **SynchronizationContext** is not ŌĆ£async itselfŌĆØ but part of how certain environments decide where continuations resume. ŅłĆciteŅłéturn11search2Ņłéturn11search21Ņłéturn7search11Ņłéturn7search5Ņłéturn7search1Ņłü

A **Claim** is a statement about an identity, not a permission by itself. The **options pattern** is typed configuration projection and validation, not just a nicer way to read JSON. **Garbage collection** is automatic managed-memory management, not deterministic resource cleanup. **Reflection** is metadata-based runtime inspection. An **assembly** is both a code container and a metadata-bearing loading unit. A **runtime identifier** describes a concrete target runtime environment for deployment, while a **target framework** describes the API surface you compile against. ## Capstone projects that actually prove competence

The **intermediate capstone** should be a production-style ASP.NET Core Web API on modern .NET using host-based startup, configuration and typed options, structured logging, EF Core with migrations, authentication and authorization, integration tests with `WebApplicationFactory`, and a real deployment target such as Azure App Service or a container. What to measure: startup health, request latency, generated SQL quality, test reliability, and deployment repeatability. What proves mastery: you can explain the request pipeline, the service graph, the DbContext lifetime, the auth flow, and the deployment mode without guessing.
