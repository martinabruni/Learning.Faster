Core Distinctions

Three things to get right first

These splits trip up nearly every developer new to the ecosystem. Get them right and everything else clicks into place.

🧩

Platform vs App Model

.NET is the platform: SDK, runtime, CLI, BCL, host, tooling. ASP.NET Core, Blazor, and .NET MAUI are app models built on top of it — not the platform itself.

🔧

SDK vs Runtime

The SDK is what you install to build: CLI, templates, build logic, and a runtime. The Runtime alone runs framework-dependent apps. Upgrading .NET starts with installing a newer SDK.

🛡️

LTS vs STS

LTS = 3 years of support. STS = 18 months. Production systems live longer than demos. Learn on .NET 10 first, understand .NET 9 as a bridge, reference .NET 8 for context still in use.

Memory Model

Stack, Heap & the Garbage Collector

🗂️ Stack vs Heap
Stack — fast, auto-cleaned when scope exits. Stores value types (int, bool, struct) and references.
Heap (Gen 0) — short-lived allocations land here. GC collects frequently.
Heap (Gen 1) — objects that survived Gen 0. Collected less often.
Heap (Gen 2) — long-lived objects (static data, caches). Expensive to collect.
Large Object Heap (LOH) — arrays ≥ 85 KB. Always Gen 2. Allocation pressure matters here.
🗑️ GC vs Dispose — two different things
✅ GC handles
Managed memory. Automatic. Non-deterministic. You don't call it — it decides when.
🔧 Dispose / using handles
File handles, DB connections, sockets, unmanaged resources. Deterministic. You call it explicitly — or use using.
⚠️ Common mistake:
Assuming GC handles file/connection cleanup. It doesn't — IDisposable does. GC is not a resource manager, it is a memory manager.
Span<T> & ArrayPool
Avoid heap allocations in hot paths. Span<T> lives on the stack; ArrayPool<T> reuses buffers.
Concepts people use without really understanding

Eight things that look simple but aren't

Each of these has a surface-level understanding and a real understanding. The gap between them is where bugs live.

Host The object responsible for startup, lifetime, DI, configuration, and logging. Every modern .NET app type runs inside a host — not just web apps.
Service Lifetime Singleton, Scoped, Transient are ownership statements. Injecting a Scoped service into a Singleton is a common bug that causes stale state.
DbContext A unit-of-work object, not a global repository. It is designed to be short-lived. Using it as a Singleton causes incorrect change tracking behaviour.
IQueryable vs IEnumerable IEnumerable = in-memory iteration. IQueryable = expression tree a provider translates to SQL. Calling LINQ on the wrong one silently fetches everything into memory.
Task ≠ Thread A Task is a promise of a result. A Thread is an OS execution resource. async/await reuses threads via the ThreadPool — it doesn't create one per request.
Claim ≠ Permission A Claim is a statement about an identity (e.g. email=x). Authorization Policies decide whether a claim grants access — that's a separate evaluation step.
Options Pattern Typed configuration projection with validation at startup. It's not just "a nicer way to read JSON" — it gives you change tokens, IValidateOptions, and named options.
Assembly Both a deployable code container and a metadata-bearing unit that the CLR loads. Reflection, AOT trimming, and DI all operate over the metadata inside assemblies.
Middleware A function that receives a context and a next delegate. It can short-circuit the pipeline or pass through. Order of registration = order of execution.
LINQ & EF Core

IQueryable vs IEnumerable — not the same

❌ Fetches EVERYTHING first
IEnumerable<User> users = db.Users;
// ↑ All rows loaded into memory
var result = users
  .Where(u => u.IsActive)
  .Take(10);
// Filter happens in C#, not SQL
✅ SQL is built and sent to DB
IQueryable<User> query = db.Users;
// ↑ No DB call yet — just an expression
var result = query
  .Where(u => u.IsActive)
  .Take(10);
// SELECT TOP 10 … WHERE IsActive = 1
Rule: keep the query as IQueryable until you call .ToList(), .FirstOrDefault(), etc. Only materialise when you actually need the data. For read-only reads, add .AsNoTracking().
Security

Authentication vs Authorization — two separate steps

🪪

Authentication — Who are you?

Establishes identity. Runs before authorization. Produces a ClaimsPrincipal attached to the request. Can use Cookies, JWT Bearer, OpenID Connect, API Keys — all pluggable via middleware.

Cookies JWT Bearer OIDC OAuth 2.0
🛂

Authorization — What can you do?

Evaluates access after identity is known. Can use Roles, Claims-based Policies, or Resource-based checks. [Authorize] attributes and IAuthorizationService both hook into the policy engine.

Publishing modes — the trade-offs

Mode Runtime required? Output size Startup Reflection / dynamic code
Framework-dependent (FDD) ✅ Must be installed on host Smallest Normal JIT Full support
Self-contained ❌ Runtime bundled Large Normal JIT Full support
Single-file Depends on FDD/SC choice One .exe file Slight overhead Full support
Native AOT ❌ Everything compiled ahead of time Medium ⚡ Instant ⚠️ Restricted — no runtime code generation
Native AOT trade-off: fastest startup and smallest memory footprint, but dynamic features (full reflection, Assembly.LoadFile, runtime code generation) are unavailable. Trimming analysis must pass. Great for containers and serverless; challenging for plugin architectures or heavy serialisation.
Watch Out

Common junior traps

These are not hypothetical — they show up in real code reviews. Each one has a real behaviour impact, not just a style concern.

DI without understanding lifetimes

Injecting a Scoped service (e.g. DbContext) into a Singleton captures it for the app's lifetime. The Singleton outlives the scope — stale state, wrong request context, data leakage between requests.

async/await with blocking calls

Calling .Result or .Wait() on a Task inside async code blocks the ThreadPool thread. In ASP.NET Core, this can starve the ThreadPool and cause severe latency spikes under load.

Treating LINQ as always in-memory

LINQ over an IQueryable is translated to SQL. Calling non-translatable methods (e.g. a C# function in a Where clause) causes a runtime exception or loads all data into memory first.

Ignoring EF Core SQL output

EF Core generates SQL you may not expect. N+1 queries from missing Include. Full-table scans from untranslated predicates. Always check the generated SQL with logging or a profiler.

"The GC handles memory" — so I'll ignore it

GC handles managed memory but you still pay allocation cost. In hot paths, allocating in every loop creates GC pressure. Use Span<T>, ArrayPool, structs, and StringBuilder appropriately.

Hard-coding secrets

Committing connection strings, API keys, or tokens to git is a real production incident waiting to happen. Use dotnet user-secrets in dev and Azure Key Vault (or environment variables) in prod.

Clean Architecture as ceremony

Layered architecture has a cost: indirection, more files, mapping between layers. Apply it when the complexity it buys down exceeds the complexity it introduces — not by default for every project.

Shipping without observability

No structured logging, no health checks, no metrics or traces = you're debugging production with Console.WriteLine. OpenTelemetry, ILogger, and health endpoints cost little and save a lot.

Proof of Competence

Capstone projects that actually prove mastery

A project proves mastery when you can explain why every part behaves the way it does — not just that it works.

⭐ Intermediate

Production-Style ASP.NET Core API

A real Web API on modern .NET with all production concerns wired in from the start.

  • Host-based startup with configuration & typed options
  • Structured logging (Serilog / built-in)
  • EF Core with migrations
  • Authentication & authorization
  • Integration tests with WebApplicationFactory
  • Real deployment target (Azure App Service or container)
Proves mastery when: you can explain the request pipeline, service graph, DbContext lifetime, auth flow, and deployment mode without guessing.
⭐⭐ Advanced

Modular Monolith or Small Distributed App

A background worker, cache or broker, resilience policies, and real observability.

  • Background worker with IHostedService
  • Redis cache or message broker integration
  • Polly-based resilience (retry, circuit breaker)
  • OpenTelemetry traces, metrics, logs
  • Containerised local dev (Docker Compose)
  • CI/CD via GitHub Actions or Azure Pipelines
Proves mastery when: you can debug the system from telemetry alone — without reading source code.
⭐⭐⭐ Expert

Performance-Sensitive .NET Subsystem

High-throughput parsing or processing with profiling and allocation analysis.

  • Pooled buffers, Span<T>, Memory<T>
  • BenchmarkDotNet with allocation tracking
  • Custom diagnostics or source generator
  • JIT vs Native AOT comparison
  • Trimming compatibility verification
  • GC pressure analysis with dotnet-trace
Proves mastery when: you can explain why each optimisation exists, what evidence justified it, and what trade-off it introduced.
